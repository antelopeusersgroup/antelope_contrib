display_ensemble

Usage:

display_ensemble [-binary -pf pffile] < infile

Algorithm:

This program is used to make a graphical display of a single ThreeComponentEnsemble object.   It is intended strictly as a quick look program.  Data are always displayed at true amplitude and for the full duration of the waveform segments in the input ensemble.  Use a gain function and/or time windowing to improve graphical display if necessary. 

The graphical interface is multithreaded so each of the three windows is active.   That means each window can be zoomed independently with MB1 in the same manner as suxwigb and dbxcor.  Typing X in any of the windows or selecting Control->Exit Event Loop will make that window no longer active.   That same action in the component 2 window will cause the program to exit with all 3 windows destroyed.  Alternatively pushing the window destroy button (red button on Mac or an x in the upper right corner of most linux window managers) in any of the 3 windows will force an exit and is harmless. 

The program has several frozen features that may be addressed with future development:
  1. The display is three separate windows with one window for each component.  The title on the top of each window is frozen as:  ThreeCEnsemblePicker:  component 0, ThreeCEnsemblePicker:  component 1, and ThreeCEnsemblePicker:  component 2.  That title is bogus as this program cannot be used for picking.   The display was simply implemented with a display object with the capability.   Limited picking may be added later.
  2. An additional consequence of using the picker graphic is that the gui contains a useless menu on the top bar "Pick Times".  Selecting anything there will only change which window will invoke a program exit via the X button or Control->Exit Event Loop.
  3. Trace labels are limited to position of each member in the ensemble (i.e. 0, 1, 2, ..., N). 
  4. The display is implemented with X-windows Motif using the seisw widget.  This is the same graphical interface used in the antelope contrib program dbxcor.  That means the only display option is shaded a shaded wiggle trace direction comparable to the seismic unix program suxwigb (seisw was derived from the CWP graphics library so the relationship is more than visual). 

Options:

-binary
Write the output as a binary serialization file.

-pf pffile
Use pffile as the parameter file in preference to the default name of display_ensemble.pf.

Pf file:

See the man page for dbxcor for a description of the graphical parameters in the default pf file.   In general, I do not recommend changing any of the default parameters other than clip_percent which can be used to set the display gain.  The display uses the same clip method (odd as it is) as suxwigb.  That is, the graphic display gain is set so that 100-clip_percent of the data samples are clipped and not displayed properly.   Seismic Unix is the only package I've ever seen use this algorithm.   It's use here is a legacy of the seisw widget's origins from CWP libraries.
