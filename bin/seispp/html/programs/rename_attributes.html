rename_attributes

Usage:

rename_attributes < in > out [-t object_type -v --help -text -pf pffile]

Algorithm:

A novel feature of the seispp library is that the namespace used for defining header (Metadata) attributes is not fixed.   The keys used to access any attribute related to a data object can be anything the user wants it to be.   Although general that complicates the problem of mapping to an external namespace such as that used by seismic unix required for a program like export_to_su.   The purpose of this program is to provide a map one namespace to another.   (e.g. rename_attributes < infile | rename_attributes | export_to_su outfile.su). 

Attributes can be either copied so the original name tag is retained, or moved (renamed).  

This program will throw an error message to stderr and blunder on if a requested attribute is not found in the header of an object in the input file.    To avoid awkward two pass processing with tons of error messages on one pass or the other, always set attributes to be copied  (not renamed) unless you are sure every single object has an attribute defined.     For this reason the general guidance is to normally copy any attribute rather than rename it as the increase in file size for copying is negligible unless the edit definition is large.

Options:

-t objt
Set the object type of data file to process.  Default is ThreeComponentSeismogram.   Supported type are:  TimeSeries, TimeSeriesEnsemble,  and ThreeComponentEnsemble.  Note that for ensembles the edits are made to the ensemble metadata, NOT individual seismogram inside the ensmble.

-text
Write the output in text format.  Default is a binary serialized file.

-v
Be more verbose.  Note all log information goes to stderr so it is best to use this option only for debugging problems.

--help
Echo the usage line. 

-pf pffile
Use pffile instead of default rename_attributes.pf.

Parameter file:

What attributes are changed in the processing is completely controlled by the parameter file. The parameter file requires only a single &Tbl entry with the tag edit_list.   Each line of the edit_list Tbl& is assumed to hold three string tokens separated by white space:
mode  oldname newname type
The mode entry must be either copy or rename.  The program will exit with an error message if this token is anything else.   If mode is set to rename the original entry with the old key is deleted.  In copy mode a second copy of the attribute is posted to the object.

oldname and newname are the old and new keys used to defined the requested attribute.   In copy mode the output will have two copies of the attribute accessible with both oldname and newname.  In rename mode only newname will be present in the output.  

The type token defines the data type of the associated attribute.   Only basic data types are allowed.   For integer attributes this field must be one of the following:  int, INT, or integer.  For floating point attributes this field must be one of the following:  real REAL, float or double.   For strings  this field must be one of the following:  string or STRING.   Boolean attributes are defined by one of the following:  bool, boolean, or BOOL.
